## 1. プロジェクト概要

### 1.1 目的

文部科学省「AI for Science 萌芽的挑戦研究創出事業（SPReAD）」の申請者を対象に、
Web UI から対話的に情報を入力し、申請書（研究計画調書）を作成・ダウンロードできる
支援システムを提供する。バックエンドの LLM は設定ファイルで切替可能とし、
Docker イメージとして配布する。

### 1.2 対象ユーザー

- SPReAD-1000 に応募する研究者（大学教員、研究員、博士課程学生等）
- AI for Science に関心があるが、AI 活用の具体的アプローチが不明な研究者

### 1.3 システム概要

`@nahisaho/spread1000-builder` のワークフローのうち、**申請書作成に必要なフェーズ
（Pre → Phase 0〜3c）** を Web UI で実現する。ウィザード形式で段階的に情報を収集し、
LLM を活用して研究プラン策定・Azure 設計・コスト算出・申請書生成・レビューまでを
一貫して支援する。

## 2. 機能要件

### 2.1 ウィザードワークフロー

#### REQ-SA-001: ウィザード全体フロー

**種別**: UBIQUITOUS
**優先度**: P0

**要件**:
THE システム SHALL spread1000-builder のワークフロー（Pre → Phase 0〜3c）に対応する
ステップバイステップのウィザード UI を提供する。

**受入基準**:
- [ ] 以下のステップが順番に表示される: コンテキスト収集 → 研究プラン → Azure設計 → コスト算出 → 申請書作成 → 申請書レビュー → 最終レビュー
- [ ] 各ステップの進捗状態（未着手/進行中/完了）が視覚的に表示される
- [ ] 完了したステップに戻って内容を修正できる
- [ ] 現在のステップと全体の進捗率が常に表示される
- [ ] レビューステップ（Phase 3b/3c）は申請書作成完了後にのみ遷移可能

**トレーサビリティ**: DES-SA-001
**パッケージ**: `spread1000-advisor`

#### REQ-SA-002: コンテキスト収集ステップ

**種別**: EVENT-DRIVEN
**優先度**: P0

**要件**:
WHEN ユーザーがウィザードを開始した場合、THE システム SHALL 6要素（PURPOSE, TARGET,
SCOPE, TIMELINE, CONSTRAINTS, DELIVERABLES）を1問ずつ対話的に収集し、
構造化メタプロンプトを生成する。

**受入基準**:
- [ ] spread1000-context-collector の 6 要素を1問1答形式で収集できる
- [ ] 各質問に対してプレースホルダーまたは入力例が表示される
- [ ] 「わからない」場合に推定値を提案し確認を求める
- [ ] 収集完了後、メタプロンプトのプレビューと承認UIが表示される
- [ ] 承認後に次のステップ（研究プラン策定）へ遷移する

**トレーサビリティ**: DES-SA-002
**パッケージ**: `spread1000-advisor`

#### REQ-SA-003: 研究プラン策定ステップ

**種別**: EVENT-DRIVEN
**優先度**: P0

WHEN ユーザーがコンテキスト収集を完了し承認した場合、THE システム SHALL メタプロンプトを
基に LLM を呼び出して AI 活用研究プランを生成し、編集・承認 UI を提供する。

**受入基準**:
- [ ] メタプロンプトを基に LLM が研究プランを生成する
- [ ] 生成された研究プランの各セクションを個別に編集できる
- [ ] 「再生成」ボタンで LLM に再度生成を依頼できる
- [ ] 承認後、`phase0-research-plan.md` として保存される

**トレーサビリティ**: DES-SA-003
**パッケージ**: `spread1000-advisor`

**種別**: EVENT-DRIVEN
**優先度**: P1

**要件**:

**受入基準**:

- [ ] GPU/CPU/ストレージ等のリソース仕様を表形式で表示する
- [ ] 各リソースの編集・追加・削除が可能

**トレーサビリティ**: DES-SA-004

#### REQ-SA-005: コスト算出ステップ

**種別**: EVENT-DRIVEN
**優先度**: P0

**要件**:
WHEN ユーザーが Azure 設計を承認した場合、THE システム SHALL Azure Retail Prices API
を呼び出して最新単価を取得し、コスト見積もりを生成する。

**受入基準**:
- [ ] Azure Retail Prices API から最新の単価を取得する
- [ ] リソースごとの月額・年額コストを表形式で表示する
- [ ] 直接経費の合計が 500 万円以下であることを検証する
- [ ] 間接経費（直接経費 × 30%）を自動計算する
- [ ] 価格検証状態（API取得済み / 推定値）を明示する
- [ ] 承認後、`phase2-cost-estimate.md` として保存される

**トレーサビリティ**: DES-SA-005
**パッケージ**: `spread1000-advisor`

#### REQ-SA-006: 申請書作成ステップ

**種別**: EVENT-DRIVEN
**優先度**: P0

**要件**:
WHEN ユーザーがコスト算出を承認した場合、THE システム SHALL Phase 0〜2 の成果物を
統合し、SPReAD 公募要領に準拠した申請書（研究計画調書）を LLM を用いて生成する。

**受入基準**:
- [ ] 様式1 の各セクション（I〜VI）を自動生成する
- [ ] 各セクションの文字数制限（日本語/英語）をリアルタイムで表示する
- [ ] 各セクションを個別に編集できる
- [ ] 経費欄に Phase 2 の検証済み単価を使用する（推定値使用時は警告表示）
- [ ] 承認後、`phase3-proposal.md` として保存される

**トレーサビリティ**: DES-SA-006
**パッケージ**: `spread1000-advisor`

#### REQ-SA-007: 申請書レビューステップ

**種別**: EVENT-DRIVEN
**優先度**: P0

**要件**:
WHEN ユーザーが申請書を作成完了した場合、THE システム SHALL 6審査観点に基づく
模擬ピアレビューを実行し、スコアリング結果と改善提案を表示する。

**受入基準**:
- [ ] 6 審査観点それぞれに ◎/○/△/× のスコアを付与する
- [ ] 各観点に対して具体的な改善提案を表示する
- [ ] 必須記載事項（10項目）のチェック結果を表示する
- [ ] 文字数制限の充足状況を表示する
- [ ] レビュー結果を `review-report.md` として保存する
- [ ] 改善提案に基づき申請書ステップに戻って修正できる

**トレーサビリティ**: DES-SA-007
**パッケージ**: `spread1000-advisor`

---

#### REQ-SA-008: 最終レビューステップ

**種別**: EVENT-DRIVEN
**優先度**: P0

**要件**:
WHEN ユーザーが申請書レビューを確認した場合、THE システム SHALL 全フェーズ成果物の
横断検証を実行し、総合スコア（18点満点）と提出可否判定（🟢/🟡/🔴）を表示する。

**受入基準**:
- [ ] 総合スコア（18点満点）と判定（🟢/🟡/🔴）を表示する
- [ ] フェーズ間整合性のチェック結果を表示する（Phase 0 ↔ Phase 3 等）
- [ ] 予算妥当性の検証結果を表示する（直接経費 ≤ 500万円、間接経費 = 30%）
- [ ] 対象外研究の該当チェック結果を表示する
- [ ] 改善優先度付きアクションリスト（High/Medium/Low）を表示する
- [ ] レビュー結果を `final-review-report.md` として保存する

**トレーサビリティ**: DES-SA-007
**パッケージ**: `spread1000-advisor`

### 2.2 LLM プロバイダー管理

#### REQ-SA-010: LLM プロバイダー設定

**種別**: UBIQUITOUS
**優先度**: P0

**要件**:
THE システム SHALL 以下の 4 種類の LLM プロバイダーを設定ファイルで切替可能にする:
Local (Ollama等), OpenAI API, Azure OpenAI, Claude API。

**受入基準**:
- [ ] 設定画面から LLM プロバイダーを選択できる
- [ ] 各プロバイダーに必要な接続情報（API Key, Endpoint, Model名等）を設定できる
- [ ] 設定は環境変数または設定ファイル（`.env` / `config.yaml`）で管理される
- [ ] プロバイダー切替時にアプリケーション再起動が不要
- [ ] 接続テスト機能で LLM の疎通確認ができる

**トレーサビリティ**: DES-SA-010
**パッケージ**: `spread1000-advisor`

---

#### REQ-SA-011: LLM プロバイダー抽象化

**種別**: UBIQUITOUS
**優先度**: P0

**要件**:
THE システム SHALL LLM プロバイダーごとの差異を吸収する統一インターフェースを
提供し、アプリケーション層が特定の LLM に依存しない設計とする。

**受入基準**:
- [ ] 統一された `LLMProvider` インターフェースが存在する
- [ ] 各プロバイダー（Local, OpenAI, Azure OpenAI, Claude）が同一インターフェースを実装する
- [ ] チャット補完（chat completion）API を統一的に呼び出せる
- [ ] ストリーミングレスポンスに対応する
- [ ] エラーハンドリングが統一されている

**トレーサビリティ**: DES-SA-011
**パッケージ**: `spread1000-advisor`

---

**トレーサビリティ**: DES-SA-012

## 3. 非機能要件

### 3.1 パフォーマンス

#### REQ-SA-050: レスポンス性能

**種別**: STATE-DRIVEN
**優先度**: P1

**要件**:
WHILE システムが通常負荷で稼働中、THE システム SHALL 静的ページの初期表示を
3 秒以内、API レスポンス（LLM 呼び出し除く）を 500ms 以内で返す。

**受入基準**:
- [ ] 静的ページの LCP（Largest Contentful Paint）が 3 秒以内
- [ ] 非 LLM API レスポンスが 500ms 以内
- [ ] LLM 呼び出しはストリーミングで逐次表示される

**トレーサビリティ**: DES-SA-050
**パッケージ**: `spread1000-advisor`

**種別**: UBIQUITOUS

外部ネットワークからのアクセスを防止する。

**トレーサビリティ**: DES-SA-062
**パッケージ**: `spread1000-advisor`

#### REQ-SA-063: XSS/インジェクション防止

**種別**: UNWANTED
**優先度**: P0

**要件**:
THE システム SHALL NOT Markdown プレビューで未サニタイズの HTML を実行し、
Excel 出力でセル先頭の数式文字（=, +, -, @）をエスケープしない状態を許容する。

**受入基準**:
- [ ] Markdown レンダリングで `<script>` タグ等の危険な HTML が除去される
- [ ] Excel 出力でセル値が `=`, `+`, `-`, `@` で始まる場合、先頭にシングルクォートを付与する
- [ ] LLM エンドポイント設定で `http://` / `https://` のみ許可し、内部ネットワークアドレスを検証する

**トレーサビリティ**: DES-SA-063
**パッケージ**: `spread1000-advisor`

### 3.3 障害回復・エラーハンドリング

#### REQ-SA-052: LLM 障害時のリトライ

**種別**: EVENT-DRIVEN
**優先度**: P0

**要件**:
WHEN LLM プロバイダーへの API 呼び出しが失敗した場合、THE システム SHALL
エラーメッセージを表示し、リトライまたはプロバイダー切替の選択肢を提供する。

**受入基準**:
